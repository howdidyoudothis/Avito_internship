# Avito QA internship — API tests (2.1)

Это решение для задания **2.1: тесты API**. Здесь собраны не только автотесты, но и всё, что обычно ждут от аккуратного тестового проекта: тест-кейсы, баг-репорты, E2E-сценарии, Allure и базовая настройка линтера.

## Что лежит в проекте

- `TESTCASES.md` — набор тест-кейсов по API
- `BUGS.md` — найденные дефекты с шагами воспроизведения
- автотесты на `pytest + requests`
- E2E-проверки основных пользовательских цепочек
- Allure-разметка в тестах
- настройка `ruff` для проверки и форматирования кода
- `allure-example.html` — пример артефакта отчёта

## Какие ручки покрыты

В проекте проверяются следующие API:

1. `POST /api/1/item` — создать объявление
2. `GET /api/1/item/{id}` — получить объявление по id
3. `GET /api/1/{sellerID}/item` — получить все объявления продавца
4. `GET /api/1/statistic/{id}` — получить статистику по объявлению (v1)
5. `GET /api/2/statistic/{id}` — получить статистику по объявлению (v2)
6. `DELETE /api/2/item/{id}` — удалить объявление

## Что именно покрывают тесты

Тесты разбиты на несколько групп, чтобы было проще и запускать, и разбирать результаты:

- **contract** — проверка соответствия ответов ожидаемому контракту
- **functional / e2e** — основные рабочие сценарии: создать, получить, найти, проверить статистику, удалить
- **negative** — невалидные данные, отсутствующие поля, несуществующие id
- **nonfunctional** — простые проверки времени ответа

## Быстрый старт

### 1. Создать и активировать виртуальное окружение

**Windows PowerShell**

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
```

**macOS / Linux**

```bash
python3 -m venv .venv
source .venv/bin/activate
```

### 2. Установить зависимости

```bash
pip install -r requirements.txt
```

### 3. Запустить весь набор тестов

```bash
pytest -v
```

Если нужен быстрый прогон по отдельным группам:

```bash
pytest -v -m contract
pytest -v -m negative
pytest -v -m e2e
```

## Отчёт Allure

Allure в этом проекте не обязателен для запуска тестов, но подключён как бонус и делает отчёт по прогону намного нагляднее.

### Установить Allure CLI

**Windows (scoop)**

```powershell
scoop install allure
```

**macOS**

```bash
brew install allure
```

**Linux**

```bash
sudo apt install allure
```

### Сохранить результаты тестов

```bash
pytest -v --alluredir=allure-results
```

### Сгенерировать и открыть отчёт

```bash
allure generate allure-results -o allure-report --clean
allure open allure-report
```

В проект также добавлен файл `allure-example.html` — его можно приложить к решению как пример готового отчёта.

## Проверка качества кода

В проекте используется `ruff` — и как линтер, и как форматтер.


```bash
ruff check . --fix
ruff format .
```

## Как устроен проект

```text
tests/
├── api/
│   ├── test_contracts.py
│   ├── test_functional.py
│   ├── test_negative.py
│   └── test_nonfunctional.py
└── helpers/
    ├── api_client.py
    ├── factories.py
    ├── parsers.py
    └── validators.py
```

Коротко по папкам:

- `test_contracts.py` — контрактные проверки ответов API
- `test_functional.py` — позитивные и E2E-сценарии
- `test_negative.py` — негативные кейсы
- `test_nonfunctional.py` — простые нефункциональные проверки
- `helpers/` — клиент, генерация данных, парсеры и валидаторы

## Почему тесты написаны именно так

Стенд общий для всех кандидатов, поэтому проект специально сделан устойчивым к чужим данным и параллельной работе:

- используются уникальные `sellerID` и имена объявлений
- тесты не завязаны на фиксированное число записей
- созданные сущности удаляются после выполнения
- один дефект не должен ломать весь набор каскадом

## Что считать хорошим результатом прогона

Нормальная картина для такого задания выглядит так:

- тесты проходят, если сервис ведёт себя корректно
- тесты падают, если найден реальный дефект
- падение сопровождается понятной причиной, а не просто «что-то пошло не так»

Именно поэтому в проекте отдельно вынесены баги, зафиксированные в `BUGS.md`, а сами проверки сделаны максимально независимыми друг от друга.
