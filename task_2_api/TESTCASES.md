# TESTCASES

## Подход

Для покрытия использованы:

- разбиение на классы эквивалентности
- граничные значения
- негативные сценарии
- E2E-проверки
- контрактные проверки
- нефункциональные smoke-checks
- corner-cases на идемпотентность / уникальность

## 1. Создание объявления — позитивные сценарии

### TC-001 — Создать объявление с валидным payload
**Предусловия:** нет  
**Шаги:**
1. Отправить `POST /api/1/item` с валидным телом.
2. Проверить код ответа.
3. Получить `id` созданного объявления.
4. Запросить объявление по `GET /api/1/item/{id}`.

**Ожидаемый результат:**  
Создание успешно, объявление доступно по id, поля совпадают с запросом.

### TC-002 — Создать два объявления одного продавца
**Шаги:**
1. Отправить два `POST /api/1/item` с одним и тем же `sellerID`.
2. Запросить `GET /api/1/{sellerID}/item`.

**Ожидаемый результат:**  
Оба объявления есть в списке продавца.

### TC-003 — Создать два объявления с одинаковым payload
**Шаги:**
1. Дважды отправить одинаковый `POST /api/1/item`.

**Ожидаемый результат:**  
Созданы два разных объявления с разными `id`.

### TC-004 — Одинаковые name/price/statistics допустимы у разных продавцов
**Шаги:**
1. Создать два объявления с одинаковыми `name`, `price`, `statistics`, но разными `sellerID`.

**Ожидаемый результат:**  
Оба объявления успешно создаются.

### TC-005 — Проверка граничных sellerID из рекомендованного диапазона
**Шаги:**
1. Создать объявление с `sellerID=111111`.
2. Создать объявление с `sellerID=999999`.

**Ожидаемый результат:**  
Оба объявления создаются успешно.
